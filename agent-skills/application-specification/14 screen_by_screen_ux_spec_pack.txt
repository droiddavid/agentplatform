SCREEN-BY-SCREEN UX SPECIFICATION PACK
Runtime AI Agent Platform v1

1. PURPOSE
This pack defines the user experience screen by screen for a non-technical consumer-facing AI agent platform. It assumes the master specification already established in this session. It is written so an implementation agent can build the frontend without guessing the user journey, layout, copy intent, or interaction rules.

2. GLOBAL UX PRINCIPLES
- Use plain language at all times.
- Always explain what the system is doing, why it needs something, and what happens next.
- Never expose technical internals unless the user explicitly asks for advanced detail.
- Every action with external impact must be clearly marked.
- Approval-related choices must be understandable in one glance.
- Mobile layouts must preserve the core action path, even if secondary panels collapse.
- Empty states must teach.
- Error states must explain and recover.

3. GLOBAL LAYOUT RULES
3.1 Signed-out layout
- Top nav: logo, product value statement, pricing, help, sign in, get started.
- Main area: hero, trust section, use-case cards, feature highlights, footer.

3.2 Signed-in desktop layout
- Left rail: Dashboard, Agents, Tasks, Runs, Connections, Documents, Billing, Privacy, Settings, Help.
- Top bar: page title, global search, notifications, usage meter, account menu.
- Main content: route-specific.
- Right utility rail on selected screens: approvals, context, recent activity.

3.3 Signed-in mobile layout
- Top bar with page title and overflow actions.
- Bottom nav for core areas: Home, Agents, Tasks, Runs, More.
- Slide-over drawers for context, approvals, and filters.

4. SCREEN INVENTORY AND BEHAVIOR

4.1 LANDING PAGE
Purpose:
- Explain value to a new user.
Primary user question answered:
- What does this do for me?
Sections:
- Hero with plain-language promise.
- Three flagship use cases: business, life, team of agents.
- How it works in 3 steps.
- Trust and privacy summary.
- Pricing preview.
- FAQ.
Primary CTAs:
- Get started.
- Sign in.
Acceptance:
- User can understand the product in under 30 seconds.

4.2 SIGN UP
Fields:
- Email.
- Password.
- Confirm password.
- Terms checkbox.
- Privacy checkbox.
Behavior:

Acceptance:

4.3 SIGN IN

5. EMPTY STATE RULES
Every empty state must contain:
- Explanation.
- Why this area matters.
- Primary CTA.
- Secondary learning link.

6. ERROR STATE RULES
Every error state must contain:
- What failed.
- What it means in plain language.
- What the user can do next.
- Retry where appropriate.

7. MICROCOPY GUIDELINES
- Say “agent” and “helper” more often than “model” or “runtime.”
- Say “approval” not “authorization token.”
- Say “remember this choice” not “persist rule.”
- Say “connected tools” not “integrations” in user-facing labels where possible.

8. ACCESSIBILITY REQUIREMENTS
- Keyboard reachable.
- Visible focus states.
- Labels tied to inputs.
- Graph view must have list fallback.
- Approval buttons must have explicit descriptions.

9. UX ACCEPTANCE SUMMARY
The UX pack is accepted only if:
- A non-technical user can create an agent.
- A non-technical user can understand an approval request.
- A non-technical user can follow what multiple agents are doing.
- A non-technical user can inspect memory and delete it.
- All critical flows are usable on mobile width.
