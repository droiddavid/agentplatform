DATABASE AND API CONTRACT PACK
Project: Consumer Runtime AI Agent Platform
Version: V1
Purpose: This document defines the database schema domains, major tables, key fields, relationships, enums, and external API contract expectations for implementation.

1. DATABASE PRINCIPLES
1.1 MySQL is the primary system of record.
1.2 All user owned data must include ownership references.
1.3 Secrets must be encrypted.
1.4 Run events are append oriented and serve as canonical trace.
1.5 Memory is separated by scope.
1.6 Approval records are durable and auditable.

4. API CONTRACT GROUPS
4.1 Auth
POST /api/auth/sign-up
Request: email, password, displayName optional
Response: user summary, verification state

POST /api/auth/sign-in
Request: email, password
Response: accessToken, refreshToken, user summary, onboarding status

POST /api/auth/refresh
Request: refreshToken
Response: accessToken, refreshToken

POST /api/auth/sign-out
Request: refreshToken or current session context
Response: success

POST /api/auth/forgot-password
Request: email
Response: success

POST /api/auth/reset-password
Request: token, newPassword
Response: success

POST /api/auth/verify-email
Request: token
Response: success

4.2 Me
GET /api/me
Response: user profile, plan summary, onboarding status, entitlements

PATCH /api/me/profile
Request: displayName, timezone, locale
Response: updated profile

GET /api/me/preferences
PATCH /api/me/preferences

15. CONTRACT RULES
- all timestamps use ISO 8601
- all IDs should be opaque strings or UUIDs
- pagination required for list endpoints where growth is expected
- ownership and authorization checks mandatory
- secrets excluded from DTOs
- DTOs versionable

16. ACCEPTANCE CRITERIA
Accepted only if:
- schema supports every declared feature in v1
- all contracts are implementable without guessing missing fields
- run traces, approvals, memory, and billing can all be persisted
- future expansion to teams and marketplace is not blocked by schema shape
