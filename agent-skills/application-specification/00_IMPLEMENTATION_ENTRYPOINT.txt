21: ==================================================
22: 2. CRITICAL RULE
23: ==================================================
24: 
25: The 23 numbered specification files are not meant to be read in simple numeric order from 01 to 23.
26: 
27: They must be read in the specific phased order defined below.
28: 
29: Reason:
30: Some files define product intent.
31: Some define architecture.
32: Some define schema and contracts.
33: Some define code structure.
34: Some define implementation order.
35: Some define validation.
36: 
37: If these are read out of order, the implementation agent may:
38: - design incorrect entities
39: - build incorrect APIs
40: - misunderstand runtime agents
41: - break approval logic
42: - build the wrong UI flow
43: - create drift from the product specification
44: 
45: ==================================================
46: 3. NON-NEGOTIABLE BUILD PRINCIPLES
47: ==================================================
48: 
49: 1. Agents are runtime-defined entities, not hard-coded compile-time role services.
50: 2. Approval-first external action handling is mandatory.
51: 3. The database is the primary system of record.
52: 4. Run events and approvals must be persisted before UI confirmation where relevant.
53: 5. The user is non-technical and the UI must remain plain-language and trust-building.
54: 6. Multi-agent collaboration is a core feature, not an optional later add-on.
55: 7. Privacy, memory controls, export, deletion, and auditability are required.
56: 8. The implementation must stay aligned with the full specification library.

216: 6. IMPLEMENTATION DISCIPLINE RULES
217: ==================================================
218: 
219: For every module:
220: 1. Re-read the relevant spec files before implementing.
221: 2. Create only the files required by the file-map packs or clearly implied by them.
222: 3. Do not invent alternate architecture when the specification already defines one.
223: 4. Validate against acceptance criteria before moving to the next module.
224: 5. Preserve runtime agent architecture at all times.
225: 6. Preserve approval-first behavior at all times.
226: 7. Preserve user privacy controls at all times.
227: 
228: ==================================================
229: 7. ANTI-DRIFT RULES
230: ==================================================
231: 
232: Do not do any of the following:
233: 
234: 1. Do not turn agents into hard-coded role classes as the primary model.
235: 2. Do not reduce the product to a single-agent chat app.
236: 3. Do not skip the approval engine just to make tool execution easier.
237: 4. Do not replace the event ledger with only in-memory frontend state.
238: 5. Do not omit memory, privacy, export, deletion, or audit because they seem like polish.
239: 6. Do not treat graph view and collaboration as optional decorations.
240: 7. Do not ignore seed data, because the product must feel ready on first boot.
241:

244: ==================================================
245: 
246: Before implementing a phase, ask and verify:

250: - What acceptance criteria apply?
251: - What UX text or user-facing wording applies?
252: - What migration or seed data changes are required?
253: - What tests must pass for this phase?
