00_IMPLEMENTATION_ENTRYPOINT
Runtime AI Agent Platform
Version 1
Master Reading and Execution Entry File for an AI Implementation Agent

==================================================
1. PURPOSE
==================================================

This is the single first file an AI implementation agent must read before reading any other file in the specification set.

Its job is to:
- define the correct reading order
- prevent drift
- explain why the order matters
- establish execution discipline
- ensure the implementation agent builds the platform in the right sequence

Do not begin coding before reading and understanding this file.

==================================================
2. CRITICAL RULE
==================================================

The 23 numbered specification files are not meant to be read in simple numeric order from 01 to 23.

They must be read in the specific phased order defined below.

Reason:
Some files define product intent.
Some define architecture.
Some define schema and contracts.
Some define code structure.
Some define implementation order.
Some define validation.

If these are read out of order, the implementation agent may:
- design incorrect entities
- build incorrect APIs
- misunderstand runtime agents
- break approval logic
- build the wrong UI flow
- create drift from the product specification

==================================================
3. NON-NEGOTIABLE BUILD PRINCIPLES
==================================================

1. Agents are runtime-defined entities, not hard-coded compile-time role services.
2. Approval-first external action handling is mandatory.
3. The database is the primary system of record.
4. Run events and approvals must be persisted before UI confirmation where relevant.
5. The user is non-technical and the UI must remain plain-language and trust-building.
6. Multi-agent collaboration is a core feature, not an optional later add-on.
7. Privacy, memory controls, export, deletion, and auditability are required.
8. The implementation must stay aligned with the full specification library.

==================================================
6. IMPLEMENTATION DISCIPLINE RULES
==================================================

For every module:
1. Re-read the relevant spec files before implementing.
2. Create only the files required by the file-map packs or clearly implied by them.
3. Do not invent alternate architecture when the specification already defines one.
4. Validate against acceptance criteria before moving to the next module.
5. Preserve runtime agent architecture at all times.
6. Preserve approval-first behavior at all times.
7. Preserve user privacy controls at all times.

==================================================
7. ANTI-DRIFT RULES
==================================================

Do not do any of the following:

1. Do not turn agents into hard-coded role classes as the primary model.
2. Do not reduce the product to a single-agent chat app.
3. Do not skip the approval engine just to make tool execution easier.
4. Do not replace the event ledger with only in-memory frontend state.
5. Do not omit memory, privacy, export, deletion, or audit because they seem like polish.
6. Do not treat graph view and collaboration as optional decorations.
7. Do not ignore seed data, because the product must feel ready on first boot.

==================================================
8. SELF-CHECK BEFORE CODING EACH PHASE
==================================================

Before implementing a phase, ask and verify:

- Which files in the spec set govern this phase?
- What exact entities, DTOs, routes, and endpoints are required?
- What acceptance criteria apply?
- What UX text or user-facing wording applies?
- What migration or seed data changes are required?
- What tests must pass for this phase?

If the answer is unclear, re-read the corresponding pack before coding.
