DATABASE MIGRATION PACK
Runtime AI Agent Platform
Version 1
Stack: Spring Boot, Java, MySQL, Flyway

==================================================
1. PURPOSE
==================================================

This pack defines the migration strategy, migration ordering, naming conventions,
and initial migration file plan for the production database. It is written so an
implementation agent can generate Flyway SQL migrations in a deterministic order.

Recommendation:
- Use Flyway
- Use versioned SQL migrations
- Keep one major concern per migration where practical
- Never modify old migrations after release
- Use repair only when absolutely necessary during development

==================================================
2. MIGRATION CONVENTIONS
==================================================

Recommended file naming:
- V1__create_users.sql
- V2__create_user_profile_and_preferences.sql
- V3__create_auth_support_tables.sql

Rules:
1. Use explicit primary keys.
2. Use explicit foreign keys.
3. Add indexes intentionally, not later by accident.
4. Prefer VARCHAR(26) ULID-style IDs for external entities.
5. Use DATETIME for timestamps.
6. Use JSON columns only where flexible structure is needed.
7. Add seed migrations separately from schema migrations.
8. Add repeatable migrations only for stable reference views or helper SQL if truly needed.

5. MIGRATION CONTENT GUIDANCE
==================================================

5.1 users migration guidance

Required columns:
- id VARCHAR(26) PRIMARY KEY
- email VARCHAR(255) NOT NULL UNIQUE
- password_hash VARCHAR(255) NOT NULL
- status VARCHAR(32) NOT NULL
- email_verified_at DATETIME NULL
- last_login_at DATETIME NULL
- plan_code VARCHAR(32) NOT NULL DEFAULT 'FREE'
- free_agent_uses_consumed INT NOT NULL DEFAULT 0
- is_active BOOLEAN NOT NULL DEFAULT TRUE
- created_at DATETIME NOT NULL
- updated_at DATETIME NOT NULL
- deleted_at DATETIME NULL

Add:
- INDEX idx_users_status(status)
- INDEX idx_users_plan_code(plan_code)

5.2 agent tables guidance

Use JSON columns for:
- goals_json
- capabilities_json if duplicated at row level for faster assembly
- safety_constraints_json
- output_style_rules_json

Also keep normalized capability and permission tables for filtering and policy enforcement.

5.3 run events guidance

run_events must support strict ordering per run:
- sequence_number BIGINT NOT NULL
- UNIQUE(run_id, sequence_number)

This table is critical. Index carefully:
- INDEX idx_run_events_run_created(run_id, created_at)

5.4 approval tables guidance

Approvals must support:
- pending inbox queries by user
- status-based filtering
- lookup by task and run

5.5 connection credentials guidance

encrypted_secret is required.
Never store raw secrets.
Store secret fingerprint when useful for troubleshooting.

==================================================
6. FOREIGN KEY RULES
==================================================

1. Use ON DELETE RESTRICT for core records unless cascading is explicitly safe.
2. For child tables like document_versions, ON DELETE CASCADE may be appropriate.
3. For user account deletion, rely on deletion job orchestration rather than broad FK cascade.
4. Keep deletion logic application-managed for auditability.

==================================================
7. INDEXING PLAN
==================================================

Add indexes for:
- users.email unique
- agents.user_id
- agents.status
- tasks.user_id
- tasks.status
- runs.user_id
- runs.task_id
- runs.status
- run_events.run_id + created_at
- approvals.user_id + status
- memory_items.user_id + scope
- connections.user_id + type
- usage_records.user_id + created_at
- notifications.user_id + is_read
- audit_logs.user_id + created_at

Do not over-index JSON fields unless a real query path requires generated columns later.

==================================================
8. SEED MIGRATION RULES
==================================================

1. Seed only stable curated data.
2. Use explicit IDs or deterministic IDs if desired for templates and tools.
3. Keep plan codes, template codes, and tool codes unique.
4. Make seed migrations idempotent only if you consciously design them that way for development.
5. For production Flyway, prefer immutable inserts in ordered migrations.

==================================================
9. DEVELOPMENT VS PRODUCTION NOTES
==================================================

Development:
- It is acceptable to reset the schema while iterating early.

Production:
- Never rewrite existing migrations.
- Add forward-only corrective migrations.
- Keep changelog discipline.

==================================================
10. IMPLEMENTATION CHECKLIST
==================================================

The migration pack is complete only if:
1. Every table from the schema pack has a migration home.
2. All FKs are defined.
3. All required indexes are defined.
4. Seed data is separated from schema creation.
5. The migration order allows the application to start cleanly on a blank database.
