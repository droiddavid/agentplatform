DATABASE SCHEMA PACK
Runtime AI Agent Platform
Version 1
Stack: Spring Boot, Java, MySQL, Angular
Purpose: Table by table field specification for a production ready consumer AI agent platform

==================================================
1. SCHEMA PRINCIPLES
==================================================

1. The database is the primary system of record.
2. Every user-owned record must include a user_id unless the table is global configuration.
3. Every important state transition must be traceable.
4. Secrets must never be stored in plaintext.
5. Every external action approval must be auditable.
6. Memory must be scoped and queryable.
7. Future support for teams and workspaces must be possible without destructive redesign.

Recommended conventions:
- Primary keys: BIGINT or CHAR(26) ULID. Prefer ULID string IDs for externally visible entities.
- Timestamps: created_at, updated_at.
- Soft delete where useful: deleted_at nullable.
- Boolean flags: is_active, is_archived, is_deleted, etc.
- JSON columns allowed for flexible metadata, but critical queryable fields should remain first class columns.
- Foreign keys must be explicit.
- Unique constraints must enforce ownership and identity rules.

==================================================
4. KEY RELATIONSHIPS
==================================================

1. users -> user_profiles one to one
2. users -> agents one to many
3. users -> tasks one to many
4. tasks -> runs one to many
5. runs -> run_agents one to many
6. run_agents -> agent_messages one to many as sender and receiver
7. runs -> approvals one to many
8. agents -> agent_tool_permissions one to many
9. users -> connections one to many
10. documents can belong to tasks and runs
11. usage_records can belong to runs
12. audit_logs can reference any entity type
13. memory_items can reference user, agent, and run depending on scope

==================================================
5. FUTURE-PROOFING FOR MULTI-TENANCY
==================================================

For v1 keep personal accounts only, but design with future expansion in mind.

Recommended future extension:
- Introduce workspaces table
- Add workspace_id to most user-owned entities
- Add memberships table
- Add roles and permissions table
- Preserve user ownership while enabling scoped collaboration

Because v1 is single user only, do not add workspace complexity to the first implementation unless it materially simplifies later migration.

==================================================
6. MIGRATION ORDER
==================================================

Recommended Flyway or Liquibase migration order:

1. users and identity tables
2. preferences and onboarding tables
3. templates tables
4. agents and policy tables
5. tasks and runs tables
6. messaging and shared context tables
7. approvals and consent tables
8. memory tables
