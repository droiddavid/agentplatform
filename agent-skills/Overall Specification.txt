# Master Specification

## Runtime AI Agent Platform for Consumers, Entrepreneurs, Creators, and Households

### Version 1 Production Build Specification

### Stack: Angular + Spring Boot + MySQL

### Delivery Goal: Complete, implementable, consumer ready system

This specification defines a full production grade web application with mobile responsive PWA behavior in version 1. The system enables non technical users to create, configure, approve, supervise, and benefit from one or more AI agents that can perform work, communicate with each other, use tools, request permission for external actions, remember prior approvals when allowed, and dynamically create additional agents at runtime.

This specification is written for machine consumption and implementation execution. It is prescriptive. It defines what must exist, how it must behave, how modules fit together, how the user experiences the app, how safety and consent work, how persistence is managed, how agent collaboration works, and how the implementation must be sequenced.

This specification assumes the design constraints established in this session and treats them as binding.

# Part A. Product Definition

## A1. Product Name Placeholder

Working product name: **AgentFlow Consumer**

The final brand name may change, but all code, design, and architecture should treat the product as a consumer facing AI agent platform.

## A2. Core Product Promise

A non technical user can describe work they need done. The platform can create one or more AI agents at runtime. Those agents can plan, research, write, organize, browse, connect to tools, prepare actions, request approval, and carry out approved work while the user remains in control.

## A3. Primary User Promise

The system must support three flagship user journeys from day one:

1. “I need an agent to help run my business.”
2. “I need an agent to manage my life and tasks.”
3. “I need a team of agents to research, plan, write, and act.”

## A4. Primary User Types

The interface, templates, and flows must support all of these audiences in version 1:

1. General consumers
2. Solo entrepreneurs
3. Small businesses
4. Creators
5. Family and household users

## A5. Core Principle

Agents are **runtime defined entities**, not hard coded Angular service classes. Angular services provide infrastructure, not agent identity.

## A6. Version 1 Scope

Version 1 includes:

1. Web application
2. Mobile responsive layout
3. PWA installability
4. Single personal account model
5. Wizard based agent creation
6. Natural language based agent creation
7. Template marketplace style browsing, without public user publishing
8. Multi agent collaboration
9. Peer to peer communication
10. Shared task board and memory
11. Dynamic agent spawning
12. Provider agnostic model abstraction
13. At least one local model backend and one cloud backend
14. Email integration
15. Calendar integration
16. Web browsing and search
17. Document generation
18. E commerce action capability
19. Custom API connector framework
20. Approval and remembered consent controls
21. Full audit, compliance, explainability, deletion, export, and privacy controls
22. Billing, usage metering, and subscription controls
23. Consumer ready onboarding and help flows

## A7. Out of Scope for Version 1 but Architecturally Supported

These are not implemented in version 1, but the architecture must support them later:

1. Native mobile apps
2. Team and workspace collaboration
3. Enterprise tenancy
4. White label deployment
5. Public template marketplace publishing
6. Avatar embodiment
7. 3D or animated embodied agents
8. Expanded connector ecosystem
9. Multi organization billing
10. Marketplace revenue sharing

# Part B. Product Outcomes and Success Criteria

## B1. User Outcome

A non technical user should be able to create an agent in under ten minutes and complete a real task in under fifteen minutes.

## B2. Business Outcome

The product should convert free users to paid users by demonstrating clear value within the first five agent uses.

## B3. System Outcome

Agents created from runtime input must be able to:

1. Know their own role, tools, and boundaries
2. Discover peer agents
3. Exchange structured messages
4. Break work into subtasks
5. Ask for permission before external action
6. Use remembered approvals if the user explicitly allowed that
7. Produce logs, explainability, and traceability

## B4. Acceptance Criteria at Product Level

The product is acceptable only if all of the following are true:

1. A user can sign up, sign in, and onboard without technical language.
2. A user can create an agent from a wizard.
3. A user can create an agent from natural language.
4. A user can start from a template.
5. A user can create two or more collaborating agents.
6. Agents can exchange direct messages.
7. Agents can post to a shared task board.
8. Agents can dynamically create an additional helper agent when policy permits.
9. Any external action is blocked pending approval by default.
10. The user can opt into remembered approvals at user, agent, or task scope.
11. The system logs all runs, actions, approvals, and messages.
12. The user can see why an action was proposed.
13. The user can revoke approvals and delete memory.
14. The user can connect supported tools.
15. The free tier limits use correctly.
16. Paid plans unlock local or cloud usage according to subscription rules.
17. The UI is usable on desktop and mobile browser sizes.
18. The product can be deployed locally first and later configured for web hosting.

# Part D. User Experience Design

## D1. UX Principle

Every flow must assume the user is not technical. The UI must explain what is happening in plain language. The system must always answer these five questions:

1. What can this agent do
2. What is it doing now
3. What does it need from me
4. What happened already
5. What will happen if I approve this

## D2. Creation Paths

The user must be able to create agents through three paths.

### D2.1 Wizard Path

The wizard asks simple questions:

1. What do you want help with
2. Is this for life, work, business, family, or creating content
3. What do you want the agent to be responsible for
4. What tools should it be allowed to use
5. Should it work alone or with other agents
6. Do you want to approve every external action
7. Should the system remember some approvals
8. What tone or style should it use
9. How proactive should it be
10. Which model preference should it use if multiple are available

### D2.2 Natural Language Path

The user can type something like:
“Create an agent that helps me manage my Etsy store, drafts responses, tracks orders, researches ideas, and asks before contacting customers.”

The system parses this, proposes one or more agents, and shows a structured review.

### D2.3 Template Path

The user can select from curated templates such as:

1. Household organizer
2. Small business assistant
3. Creator research assistant
4. Document and email agent
5. Calendar and planning assistant
6. E commerce operations helper
7. Multi agent research team

## D3. Review Before Creation

Regardless of entry path, the user must review:

1. Agent name
2. Role
3. Capabilities
4. Allowed tools
5. Approval behavior
6. Memory behavior
7. Whether it can communicate with peers
8. Whether it can spawn helper agents
9. Model preference
10. Estimated type of tasks it can handle

## D4. Main Workspace UX

The main workspace for multi agent runs must contain:

1. Left panel for agents and task board
2. Center panel for chat or run activity
3. Right panel for approvals, context, and documents
4. Toggle for graph view
5. Clear banner showing current run status
6. Clear badge showing whether action is blocked awaiting approval

## D5. Approval UX

Approvals must be easy to understand. For every proposed external action, the user sees:

1. What the action is
2. Why the agent wants to do it
3. What data will be sent
4. What tool or service will be used
5. Whether the action can be remembered next time
6. Options to approve once, always approve in similar cases, deny, or revise instruction

1. Agent identifies a missing specialty
2. Agent submits spawn proposal
3. System checks agent and user policy
4. If spawn requires approval, user is asked
5. If approved, system creates runtime child or sibling agent from structured spec
6. New agent enters run with limited or inherited context
7. Run log records spawn reason and inputs

## E10. External Actions

All external actions require approval by default.

### Examples

1. Send email
2. Create calendar event
3. Submit external API request
4. Publish or send generated document
5. Update e commerce listing or response
6. Call a remote service

### Approval memory options

1. Approve once
2. Approve for this task
3. Approve for this agent on similar actions
4. Approve for this user on similar actions
5. Never allow this pattern
6. Ask every time

## E11. Memory

Version 1 supports all of the following with privacy controls:

1. Per run memory
2. Per agent long term memory
3. User profile memory
4. Shared memory across cooperating agents

Primary persistence must be stored in the database. Summaries and memory items should be structured and queryable. Large blobs or generated files may live in object or file storage, but records and references live in the database.

# Part F. Non Functional Requirements

## F1. Performance

1. Initial app shell should load fast on consumer broadband and modern phones.
2. Major screens must use lazy loading.
3. Agent activity views must update progressively.
4. Long running actions must stream status to the UI.

## F2. Reliability

1. Failed tool calls must be retriable.
2. Runs must resume gracefully after transient backend interruption where possible.
3. All important events must be persisted before user facing completion status is shown.

## F3. Security

1. Passwords must be hashed.
2. Tokens must be rotated.
3. Sensitive connection tokens must be encrypted at rest.
4. Audit logs must be immutable or append only in behavior.
5. Approval events must be non repudiable in the product record.

## F4. Privacy

1. User controls memory
2. User controls approval persistence
3. User can export data
4. User can delete data
5. User can revoke tool connections
6. System explains what is stored and why

## F5. Accessibility

1. Keyboard friendly
2. Screen reader support
3. Sufficient contrast
4. Error messages in plain language
5. Touch friendly on mobile

## F6. Explainability

Every meaningful agent action must be explainable in user readable form.

## F7. Compliance Foundations

Version 1 must include:

1. Audit trail
2. Consent records
3. Data retention settings
4. Data export
5. Data deletion
6. Terms acceptance record
7. Privacy notice acceptance record

# Part K. Agent Runtime Specification

## K1. Agent Definition Object

Each runtime agent must be represented by a structured spec containing:

1. Identity fields
2. Role definition
3. Goals
4. Capabilities
5. Tool permissions
6. Approval policy
7. Memory policy
8. Collaboration policy
9. Spawn policy
10. Model routing preference
11. Prompt scaffold
12. Safety constraints
13. Output style rules

## K2. Capability Model

A capability is not a TypeScript class. It is a declared ability. Examples:

1. research
2. summarize
3. plan
4. draft
5. critique
6. schedule
7. communicate
8. analyze
9. shop support
10. commerce support
11. api action

Capabilities may map to tool rights, prompt strategy, or workflow permissions.

## K3. Tool Permission Model

Each tool permission must define:

1. Tool id
2. Access mode
3. Approval requirement
4. Allowed data categories
5. Rate or usage limit
6. Scope
7. Notes

## K4. Collaboration Policy

Each agent must define:

1. Whether peer messaging is allowed
2. Whether shared memory is allowed
3. Which peer types or specific peers may be contacted
4. Whether broadcast is allowed
5. Whether critique loops are allowed
6. Whether spawn requests are allowed

## K5. Runtime Turn Cycle

Each active turn follows this sequence:

1. Receive current context
2. Load self identity and constraints
3. Load relevant memory
4. Load peer directory summary
5. Decide next action
6. If internal only, produce output or message
7. If external action intended, create approval request
8. If tool call allowed and approved, execute tool
9. Record result
10. Update memory summary
11. Emit run event
12. Determine next actor or finish

## K6. Self Awareness and Peer Awareness

Software level awareness means:

1. Agent knows who it is
2. Agent knows what it may do
3. Agent knows its current task
4. Agent knows what peers exist
5. Agent knows what peers are advertised to do
6. Agent knows how to ask peers for help

This must be implemented through structured context injection, not anthropomorphic claims.

## K7. Spawn Model

When spawning occurs, the new agent must be created from a spec generated from current task need. Inherited properties must be explicitly controlled. By default, the new agent inherits:

1. User ownership
2. Current run id
3. Parent run memory access only if allowed
4. Limited lifespan unless saved by user
5. Parent approval boundaries or stricter

## K8. Run Completion Criteria

A run completes only when:

1. Task goal reached
2. Final summary generated
3. Pending approvals resolved or dismissed
4. Outputs saved
5. Usage recorded
6. User notified
